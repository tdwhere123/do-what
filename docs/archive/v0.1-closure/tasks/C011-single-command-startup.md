# C011 — 收口为单入口启动或等价编排启动

**优先级：** P0（必须，当前启动方式不符合可用产品预期）
**涉及范围：**
- 根目录 `package.json`
- 如需新增编排脚本：`scripts/` 或现有启动入口
- `packages/app/src/main/`（如需要主进程接线）
- `packages/core/src/server/`（如需要配合启动编排）
- `README.md`

---

## 背景

当前主验收路径要求用户分别理解并手动执行：

1. 先启动 Core
2. 再启动 App
3. 再观察引擎 / 模块是否真的接上

这对开发调试可以接受，但对 v0.1 作为产品验收入口不可接受。至少在仓库主路径中，应该提供一个单入口，把必要模块按顺序拉起。

---

## 目标

1. 为 v0.1 提供单入口启动路径
2. 保留分步命令用于调试，但不再把分步手工拼装当成默认产品入口
3. 让 README、任务卡、人工验收路径与真实启动方式一致

---

## 决策

本任务采用以下默认方案：

- **开发与验收主路径：** 增加根命令 `pnpm dev`，由它负责编排 Core 与 App 的启动顺序
- **调试子路径：** 继续保留 `pnpm dev:core` 和 `pnpm dev:app`
- **v0.1 不要求：** 一次性解决打包后系统级守护进程或安装器级别的自启动

也就是说，v0.1 的最低可用要求是“从仓库本地启动时，不再需要手动分步理解主路径”。

---

## Core 健康等待策略（必须遵守）

- **工具选择：** 使用 `wait-on`（`pnpm add -D wait-on`）等待 `http://127.0.0.1:3847/health`
- **超时：** 最多等待 30 秒，超时则打印明确报错并退出，不静默挂起
- **实现参考：**
  ```json
  "dev": "concurrently --kill-others-on-fail \"pnpm dev:core\" \"wait-on http://127.0.0.1:3847/health --timeout 30000 && pnpm dev:app\""
  ```
  或等价的 shell script，核心是：轮询 `/health` → 成功后启动 App → 超时报错退出
- 不允许用 `sleep N` 替代健康检查

## 本任务必须完成

1. 新增单入口启动命令 `pnpm dev`
2. 单入口需要负责最小编排：
   - 拉起 Core
   - 等待 Core 进入可连接状态（使用 `wait-on`，见上方策略）
   - 等待成功后启动 App
3. 若 C010 已实现 engine 默认接线，则单入口需要纳入该默认流程
4. `README.md` 的快速开始必须改为单入口优先，分步命令降级为“调试用”
5. 人工验收说明必须改成以单入口为默认路径，而不是继续要求两终端手工拼装

---

## 本任务不包含

- 不要求完成安装器、后台常驻服务、开机自启
- 不要求一次兼容所有未来平台差异
- 不要求一次解决 packaged release 的系统集成问题

---

## 验收标准（DoD）

1. 仓库根目录存在单入口启动命令
2. 使用该命令可以完成 v0.1 主路径启动，不再需要先记住多个步骤再手动组合
3. `README.md` 与 `closure-overview.md` 中的主路径启动说明一致
4. `dev:core` / `dev:app` 仍保留，但被明确标记为调试路径

---

## 人工严苛验收标准

- [ ] 仓库根目录 `package.json` 存在 `pnpm dev`（或等价命令）
- [ ] 执行 `pnpm dev` 后，Core 和 App **都能自动启动**，无需手动分步操作
- [ ] `pnpm dev` 使用 `wait-on`（或等价机制）等待 Core 就绪，**不是简单 `sleep`**
- [ ] Core 30s 内未就绪时，有清晰报错退出，不是静默挂起
- [ ] `README.md` 快速开始已更新为单入口优先，分步命令已降级为”调试用”
- [ ] 原有 `dev:core` / `dev:app` 仍保留且可单独使用
